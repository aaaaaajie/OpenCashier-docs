# Contributing to OpenCashier

OpenCashier is still pre-1.0. Small, focused pull requests are easier to review and much less likely to cause payment-flow regressions.

This page explains how external contributors should prepare a change, when a changelog entry is required, and which parts of the changelog flow are automated today.

## Before you start

- Check existing issues and pull requests before starting similar work.
- For large features, architecture changes, or provider-level integrations, open an issue first to align on scope.
- Never commit real payment keys, certificates, merchant secrets, or `.env` files.
- Preserve existing provider behavior unless the change explicitly updates that contract.

## Local development

1. Install dependencies.

```bash
pnpm install
```

2. Start local infrastructure.

```bash
docker compose -f docker-compose.infrastructure.yml up -d
```

3. Copy environment variables.

```bash
cp .env.example .env
```

4. Generate Prisma Client and sync the schema.

```bash
pnpm prisma:generate
pnpm prisma:db:push
```

5. Start the workspace.

```bash
pnpm dev
```

## Validation

Run the checks that match your change before opening a PR.

```bash
pnpm typecheck
pnpm smoke:merchant
```

Additional commands you may need:

```bash
pnpm build
pnpm test:tunnel
pnpm changelog:validate
pnpm changelog:draft
```

## Changelog model

OpenCashier now uses a split changelog model:

- Public changelog files at the repo root are release-only.
- Unreleased externally visible changes are stored as internal bilingual fragments under `.github/changelog/unreleased/`.
- Internal draft files are generated from those fragments.
- Maintainers promote pending fragments into a released changelog section when cutting a public tag.

This means contributors should not edit the public root changelog files directly for unreleased work.

## When a changelog fragment is required

Add a fragment when your change is externally visible, especially if it affects:

- merchant API routes, payloads, auth headers, signing, idempotency, or webhook behavior
- hosted cashier behavior, `cashierUrl`, routing, or provider session behavior
- deployment topology, Docker images, compose files, runtime config, published ports, reverse-proxy mode, or database wiring
- required or default environment variables
- provider availability, supported channels, or checkout defaults
- admin or operator workflows such as login, provider configuration, or first-run setup
- quickstart, SDK, or onboarding paths that change how integrators start
- fixes for user-visible bugs, upgrade issues, or rollback risks

Usually no fragment is needed for:

- pure refactors with no external effect
- formatting, comments, or test-only updates
- internal file moves or CI cleanup with no published artifact change
- docs wording fixes when product behavior did not change

## Contributor flow

If you contribute a code change with external impact, the expected flow is:

1. Implement the product change.
2. Add one bilingual fragment JSON file under `.github/changelog/unreleased/`.
3. Run `pnpm changelog:validate`.
4. Include the fragment in the PR together with the code change.

Example: if your PR changes merchant-visible behavior, add a file such as `.github/changelog/unreleased/2026-03-20-merchant-timeout.json`.

```json
{
  "title": "merchant-timeout",
  "entries": [
    {
      "category": "Changed",
      "en": "Changed the default merchant notification timeout to 10 seconds.",
      "zh-CN": "将商户通知的默认超时时间调整为 10 秒。"
    }
  ]
}
```

Write from the deployer or integrator point of view. Describe the external effect, not the internal refactor.

## What is automated today

Partially automated:

- `pnpm changelog:validate` validates fragment shape and draft consistency
- `pnpm changelog:draft` regenerates internal pending-release draft files
- GitHub Actions validate changelog data in pull requests
- pushes to `main` automatically refresh the internal draft files

Still manual by design:

- deciding whether a change deserves a changelog fragment
- writing the final user-facing wording
- choosing the release version and release date
- promoting pending fragments into a released changelog section

Maintainers cut a release entry with:

```bash
pnpm changelog:release -- v0.1.0-beta.5 2026-03-20
```

## Pull request checklist

- The change is scoped and explained clearly.
- A changelog fragment was added for externally visible changes.
- Documentation was updated where behavior changed.
- No secrets, production credentials, or private certificates were added.
- Relevant commands were run locally, and the results are described in the PR.

## Communication

- Bug reports: use the bug issue template.
- Feature ideas: use the feature request template.
- Usage or integration questions: use the support/question template.
- Security issues: follow the repository security policy.
