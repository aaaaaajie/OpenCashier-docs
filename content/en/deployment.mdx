# Docker Image Deployment Guide

This document defines the standard deployment for OpenCashier using official container images. Two deployment modes are supported:

- Option A: Direct deployment
- Option B: Integrate with an existing reverse proxy

Both options use the same images, the same base compose files, and the same tag strategy. The difference is only in the public entry and Docker networking.

## 1. File Responsibilities

The compose files in the repo are organized by responsibility:

- `docker-compose.infrastructure.yml`
  - Local development infrastructure
  - Starts only PostgreSQL and Redis
  - Not for server deployment
- `docker-compose.deploy.yml`
  - Base compose for image deployment
  - Starts `web`, `api`, and `postgres`
  - Suitable for all server deployment scenarios
- `docker-compose.deploy.reverse-proxy.yml`
  - Reverse proxy extension
  - Adds external Docker networks and aliases for `web` and `api`
  - Only used when the existing gateway also runs in Docker
- `.env.deploy.example`
  - Deployment environment template

## 2. Choose a Deployment Mode

Select the deployment mode based on your entrypoint topology:

| Scenario | Mode |
| --- | --- |
| Standalone OpenCashier, OpenCashier owns public ports | Option A: Direct deployment |
| Existing Nginx/Caddy/Traefik already uses `80/443` | Option B: Integrate reverse proxy |

Reverse proxy is not the default prerequisite. For first-time validation, standalone deployment, or single-system deployment, Option A is recommended.

## 3. Deployment Topology

The standard topology includes three containers:

- `web`
  - Hosts frontend static assets
  - Public web entry for OpenCashier
  - Proxies `/api/*` to `api`
- `api`
  - Merchant API
  - Admin API
  - Swagger
  - Hosted cashier entry
- `postgres`
  - Stores orders, refunds, notifications, and platform config

Internal container hostnames are fixed and unique:

- `opencashier-web`
- `opencashier-api`
- `opencashier-postgres`

This avoids name collisions in shared Docker networks.

Public endpoints are standardized as:

- Admin: `https://pay.example.com/`
- Merchant API: `https://pay.example.com/api/v1`
- Swagger: `https://pay.example.com/api/docs`
- Hosted cashier: `https://pay.example.com/api/cashier/{cashierToken}`

## 4. Runtime Requirements

Base requirements:

- Docker Engine
- Docker Compose v2
- Linux server
- Public domain

Extra requirements for Option A:

- OpenCashier can bind a public port directly

Extra requirements for Option B:

- Existing reverse proxy already exposes a public entry
- Reverse proxy and OpenCashier can join the same Docker network

## 5. Image Tag Strategy

Image tags are grouped into three types:

- Pre-release tag: `v0.1.0-beta.1`
- Stable release tag: `v0.1.0`
- Rolling tag: `latest`

Choose the image version with `OPENCASHIER_IMAGE_TAG`. Always deploy with an explicit version tag such as `v0.1.0-beta.1` or `v0.1.0`. Rollback by switching `OPENCASHIER_IMAGE_TAG` back.

The base compose does not set `pull_policy: always`. For upgrades and rollbacks, explicitly run `docker compose pull` once. Routine `docker compose up -d` does not force a pull.

## 6. Common Preparation

### 6.1 Copy deployment env file

```bash
cp .env.deploy.example .env.deploy
```

### 6.2 Set required variables

Before first deployment, set the following variables:

- `APP_SECRET`
- `PLATFORM_CONFIG_MASTER_KEY`
- `APP_BASE_URL`
- `WEB_BASE_URL`
- `ADMIN_USERNAME`
- `ADMIN_PASSWORD`
- `POSTGRES_PASSWORD`

Recommended public URL configuration:

```text
APP_BASE_URL=https://pay.example.com
WEB_BASE_URL=https://pay.example.com
APP_API_BASE_URL=/api/v1
ADMIN_USERNAME=admin
ADMIN_PASSWORD=replace-with-a-strong-admin-password
ENABLE_DEMO_DATA=0
```

The PostgreSQL connection for the built-in database is generated by the compose file, using host `opencashier-postgres`. Only set `DATABASE_URL` when connecting to an external database.

### 6.3 Admin access safety

OpenCashier separates public payment entry and admin management:

- `/api/cashier/*` and merchant signed APIs are public
- `/api/v1/admin/*` requires admin auth
- The admin UI checks session state first; without admin credentials it cannot read or modify config

If `ADMIN_USERNAME` or `ADMIN_PASSWORD` is missing, the API refuses to start. Local and production use the same admin auth flow.

To avoid demo merchants in production, keep:

```text
ENABLE_DEMO_DATA=0
```

## 7. Option A: Direct Deployment

OpenCashier exposes `web` to the public directly. Suitable for standalone servers, dedicated domains, or quick first validation.

### 7.1 Recommended env vars

```text
WEB_PUBLISHED_BIND=0.0.0.0
WEB_PUBLISHED_PORT=80
```

### 7.2 Start services

```bash
docker compose --env-file .env.deploy -f docker-compose.deploy.yml pull
docker compose --env-file .env.deploy -f docker-compose.deploy.yml up -d
```

### 7.3 Check status

```bash
docker compose --env-file .env.deploy -f docker-compose.deploy.yml ps
docker compose --env-file .env.deploy -f docker-compose.deploy.yml logs -f
```

## 8. Option B: Integrate Reverse Proxy

Use this mode when a server already has a public gateway and OpenCashier is added as another site or subdomain.

### 8.1 Preconditions

Your existing proxy should meet these conditions:

- The proxy itself runs in Docker
- The proxy and OpenCashier can join the same external Docker network
- The proxy reaches OpenCashier `web` container via network alias

If your proxy runs on the host (not in Docker), forward directly to `127.0.0.1:${WEB_PUBLISHED_PORT}` and do not use `docker-compose.deploy.reverse-proxy.yml`.

### 8.2 Recommended env vars

```text
WEB_PUBLISHED_BIND=127.0.0.1
WEB_PUBLISHED_PORT=18080
OPENCASHIER_REVERSE_PROXY_NETWORK=xxx
OPENCASHIER_API_NETWORK_ALIAS=opencashier-api
OPENCASHIER_WEB_NETWORK_ALIAS=opencashier-web
```

Variable definitions:

- `WEB_PUBLISHED_BIND=127.0.0.1`
  - `web` only listens on localhost
- `WEB_PUBLISHED_PORT=18080`
  - Local forward port on host
- `OPENCASHIER_REVERSE_PROXY_NETWORK`
  - External Docker network name for your existing gateway
- `OPENCASHIER_API_NETWORK_ALIAS`
  - Upstream name used by the gateway for `api`
- `OPENCASHIER_WEB_NETWORK_ALIAS`
  - Upstream name used by the gateway for `web`

### 8.3 Start services

```bash
docker compose --env-file .env.deploy \
  -f docker-compose.deploy.yml \
  -f docker-compose.deploy.reverse-proxy.yml \
  pull

docker compose --env-file .env.deploy \
  -f docker-compose.deploy.yml \
  -f docker-compose.deploy.reverse-proxy.yml \
  up -d
```

### 8.4 Check status

```bash
docker compose --env-file .env.deploy \
  -f docker-compose.deploy.yml \
  -f docker-compose.deploy.reverse-proxy.yml \
  ps

docker compose --env-file .env.deploy \
  -f docker-compose.deploy.yml \
  -f docker-compose.deploy.reverse-proxy.yml \
  logs -f
```

### 8.5 Reverse proxy routing requirements

For a Dockerized gateway, route inside the `cashier` site as:

- `/` -> `OPENCASHIER_WEB_NETWORK_ALIAS`
- `/api/` -> `OPENCASHIER_API_NETWORK_ALIAS`

Reasons:

- The `/` and `/api/` paths on `cashier.linkmind.site` both belong to OpenCashier
- Avoid an extra hop from `web` to `api`
- Avoid name collisions with generic `api`/`web` in shared networks
- Clear routing for frontend and API within the same domain

Example Nginx config:

```nginx
upstream opencashier_web {
    least_conn;
    server opencashier-web:80 max_fails=3 fail_timeout=30s;
    keepalive 16;
}

upstream opencashier_api {
    least_conn;
    server opencashier-api:3000 max_fails=3 fail_timeout=30s;
    keepalive 16;
}

server {
    listen 443 ssl http2;
    server_name cashier.example.com;

    ssl_certificate /etc/nginx/ssl/cashier-fullchain.pem;
    ssl_certificate_key /etc/nginx/ssl/cashier-privkey.pem;

    location /api/ {
        proxy_pass http://opencashier_api;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location / {
        proxy_pass http://opencashier_web;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

If you use host-level Nginx or do not want to split `/` and `/api/`, you can still proxy the entire site to `web`. In that case, `web` will keep forwarding `/api/*` to `api` internally.

## 9. Upgrade

After updating `OPENCASHIER_IMAGE_TAG` in `.env.deploy`, pull and restart.

Option A:

```bash
docker compose --env-file .env.deploy -f docker-compose.deploy.yml pull
docker compose --env-file .env.deploy -f docker-compose.deploy.yml up -d
```

Option B:

```bash
docker compose --env-file .env.deploy \
  -f docker-compose.deploy.yml \
  -f docker-compose.deploy.reverse-proxy.yml \
  pull

docker compose --env-file .env.deploy \
  -f docker-compose.deploy.yml \
  -f docker-compose.deploy.reverse-proxy.yml \
  up -d
```

## 10. Rollback

Set `OPENCASHIER_IMAGE_TAG` back to the target version, then run `pull` and `up -d` for your current mode.

## 11. Database

The default deployment includes PostgreSQL with a named volume `postgres-data` for persistence. This works for single-node deployments and standard image-based setups.

To use an external PostgreSQL instance, override with `DATABASE_URL`:

```text
DATABASE_URL=postgresql://user:password@db.example.com:5432/cashier?schema=public
```

After switching to an external DB, you can remove the `postgres` service from your compose file.

## 12. Database Schema Sync

By default, the API container runs:

```text
prisma db push
```

If your DB schema is managed externally, set:

```text
SKIP_PRISMA_DB_PUSH=1
```

## 13. Frontend API Runtime Config

`web` uses the following runtime config by default:

```text
APP_API_BASE_URL=/api/v1
```

This keeps API base URL relative to the same domain rather than hard-coding absolute URLs in build artifacts. To override with an absolute address:

```text
APP_API_BASE_URL=https://another-host.example.com/api/v1
```

After deployment, admins can also copy the Merchant API base and Swagger URLs from the “Merchant Apps” page, no need to infer the API path manually.

## 14. Image Publishing

Default registry is GitHub Container Registry:

- `ghcr.io/<owner>/opencashier-api`
- `ghcr.io/<owner>/opencashier-web`

Rules:

- `v*` tags build versioned tags
- Each tag build also publishes `sha-*`
- Stable version tags also publish `latest`

The automation lives in `.github/workflows/publish-images.yml`.

### 14.1 Optional China mirror

The workflow can optionally push to a mirror registry (useful for mainland China), such as Alibaba Cloud ACR Personal.

Configuration:

- GitHub Actions Variables
  - `MIRROR_REGISTRY`
    - e.g. `registry.cn-hangzhou.aliyuncs.com`
  - `MIRROR_NAMESPACE`
    - e.g. `your-namespace`
- GitHub Actions Secrets
  - `MIRROR_REGISTRY_USERNAME`
  - `MIRROR_REGISTRY_PASSWORD`
